Background

Two new global events, unhandledrejection and rejectionhandled, can be used to keep track of Promise rejections, including whether those rejections are handled after the fact.

The unhandledrejection event is the promise-equivalent of the global error event, which is fired for uncaught exceptions. Because a rejected promise could be handled after the fact, by attaching catch(onRejected) or then(onResolved, onRejected) to it, the additional rejectionhandled event is needed to indicate that a promise which was previously rejected should no longer be considered unhandled.

These two events can be used, for example, to keep track of unhandled rejections as part of an error-reporting framework.

This sample generates a new rejected promise every second. Half of the time, it catch()es that rejection after the fact to simulate a handled error, and half of the time the promise remains uncaught.

Every five seconds, a "logging" function is invoked, which reports all the promises that went uncaught. In a real-world system, this logging might report back to a centralized database which kept track of all unhandled errors, but for this example, it just logs the list of errors to the page.
